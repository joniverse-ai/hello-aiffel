## 1. 개요

### 1.1 서비스 한 줄 정의
> 기업 총무 담당자가 VIP 답례 선물을 상황별로 고르고 문의할 수 있는 B2B 셀렉트샵.

### 1.2 배경 / 이 방향을 택한 이유
- 일반 소비자 디저트 시장은 가격 경쟁이 격심하고 브랜드 없이 진입 불가.
- **답례 선물 시장은 "검색·비교"보다 "실수 없이 그럴듯한 걸 빨리"가 핵심** → UX 차별화 여지가 큼.
- 초프리미엄 소량은 자체 생산 역량과 부합, 브랜드 디자이너의 강점(시각 자산·톤)이 곧 상품력.

### 1.3 하지 않을 것 (Non-goals)
- 일반 소비자·개인 선물 시장
- 냉장 생재료 디저트 (당일 배송·짧은 유통기한)
- 카탈로그 외의 완전 커스텀 (로고 각인·박스 커스텀 등)
- 대량·저가 캠페인 답례품

## 2. 목표와 성공지표 (KPI)

### 2.1 3개월 KPI
- **누적 주문 건수** (목표치 별도 확정)
- **반복구매사 수** — 최소 5개 회사

### 2.2 성공 판단 기준
매출액은 부차 지표. **"이 브랜드에 다시 주문한 회사가 있는가"** 가 본질적 성공 여부.

### 2.3 실패 시 재검토 지점
- 반복구매율이 낮으면 → 상품 컨셉·큐레이션 재검토
- 주문 자체가 부족하면 → 인스타 콘텐츠·유입 전략 재검토

## 3. 타깃 페르소나

### 3.1 주 페르소나 — "총무팀 김대리"
- **역할**: 대기업·중견기업 총무·경영지원·인사팀 담당자
- **트리거**: 상사(임원)로부터 "이번 계약 감사 답례로 15개 준비해" 지시
- **핵심 니즈**
  - 실수하지 않기 (상대방이 VIP)
  - 상사에게 "이거 어떠세요"라고 보여줄 근거
  - 세금계산서 자동 발행
- **불편함(Pain)**
  - 매번 수신자 이름·주소를 하나씩 입력하는 노동
  - "이런 상황엔 뭐가 좋을까?"의 답을 못 찾음 (기존 사이트는 개인 선물 위주)
  - 상사 결재용 자료(가격·구성·이미지)를 스크린샷으로 뜯어야 함

### 3.2 최종 수신자 — VIP
- **역할**: 고객사 대표·임원·핵심 실무자
- **관여도**: 낮음. 잘 만든 걸 받아보고 브랜드 감을 형성.
- **함의**: 수신자용 UI는 없음. **포장·카드·박스가 브랜드 첫인상**.

## 4. 사용자 시나리오

### 시나리오 A: 계약 감사 답례 15건 (평상시)
1. 총무 김대리가 인스타에서 브랜드를 이미 팔로우 중.
2. 프로필 링크 → 사이트 홈 진입.
3. 홈에서 Gift 섹션의 **"계약 감사"** 탭 클릭.
4. 상황에 맞는 세트가 카드 캐러셀로 나옴 (가격·구성 명확).
5. 하나 선택 → 모달 상세에서 구성·가격 확인 → 문의하기.
6. 문의 폼 작성 → Supabase에 저장.
7. 운영자가 문의 확인 후 직접 연락하여 주문 진행.

### 시나리오 B: 명절 사전예약 30건
- Gift 섹션 "명절 인사" 탭 클릭 시 사전 예약 안내 배너 노출.
- 이후 흐름은 시나리오 A와 동일 (문의 기반 주문).

## 5. 상품 및 비즈니스 모델

### 5.1 상품
- 수제 티·커피 페어링 세트 (상온 보관 박스)

### 5.2 상황 카테고리 — 3개 (구현 완료)
| 카테고리 | 세트 |
|---|---|
| **계약 감사** | 가온, 온율, 소담, 예림, 윤하 |
| **임원 축하** | 수안, 담헌, 정연, 설아, 채윤 |

### 5.3 상품 필터
- 전체 / 티 세트 / 커피 세트 / 페어링 — 서브탭으로 구현 완료

### 5.4 주문 규모·가격대
- 소량 초프리미엄 — **1회 5~30건, 건당 10~30만원**

### 5.5 리드타임·MOQ (이원화)
| 모드 | MOQ | 리드타임 | 사이트 안내 |
|---|---|---|---|
| 평상시 | 5건 | 7영업일 | 상세 페이지 내 명시 |
| 명절 시즌 | 10건 | 사전 예약 (2~4주 전 마감) | Gift 섹션 명절 탭 내 배너 |

### 5.6 배송
- **개별 발송만** (수신자별 각각의 주소)

### 5.7 결제 (현재)
- 문의 기반 수동 처리 (프로토타입 단계)
- 향후 PG 연동 검토

## 8. 기술 스택

### 8.1 현재 구성 (프로토타입)

| 레이어 | 선택 |
|---|---|
| 프론트엔드 | Static HTML/CSS/JS (프레임워크 없음) |
| 백엔드/DB | **Supabase** (PostgreSQL) — 서울 리전 (ap-northeast-2) |
| 인증 | **Supabase Auth** (이메일+비밀번호) |
| 호스팅 | **GitHub Pages** (joniverse-ai/danjeong) |
| 배포 | danjeong-deploy 폴더 → git push → GitHub Actions |
| CDN | Supabase JS SDK (jsdelivr CDN) |

### 8.2 Supabase 데이터베이스 스키마

**profiles** (회원 프로필)
| 컬럼 | 타입 | Nullable | 기본값 |
|---|---|---|---|
| id | uuid (PK, FK → auth.users) | NO | — |
| company | text | YES | — |
| name | text | YES | — |
| phone | text | YES | — |
| created_at | timestamptz | YES | now() |

**inquiries** (상품 문의)
| 컬럼 | 타입 | Nullable | 기본값 |
|---|---|---|---|
| id | (PK) | NO | auto |
| product_name | text | — | — |
| company | text | — | — |
| name | text | — | — |
| email | text | — | — |
| phone | text | YES | — |
| quantity | integer | YES | — |
| message | text | YES | — |
| user_id | uuid (FK → auth.users) | YES | — |
| created_at | timestamptz | — | now() |

### 8.3 RLS 정책

**profiles:**
| 정책 | 동작 | 조건 |
|---|---|---|
| 본인 프로필 조회 | SELECT | auth.uid() = id |
| 본인 프로필 생성 | INSERT | auth.uid() = id |
| 본인 프로필 수정 | UPDATE | auth.uid() = id |

**inquiries:**
| 정책 | 동작 | 조건 |
|---|---|---|
| 누구나 문의 등록 | INSERT | true (비로그인도 가능) |
| 본인 문의 조회 | SELECT | auth.uid() = user_id |

### 8.4 트리거
- `on_auth_user_created` → `handle_new_user()`: 회원가입 시 profiles 자동 생성 (SECURITY DEFINER)

### 8.5 향후 확장 계획

Phase 2에서 주문 시스템이 필요해지면:
- **orders** 테이블 추가 (주문 정보)
- **order_items** 테이블 (수신자별 배송 정보)
- PG 연동 (토스페이먼츠 등)
- 세금계산서 자동 발행 (팝빌 등)

아임웹 대신 Supabase 기반으로 직접 구축하는 방향으로 전환됨.
